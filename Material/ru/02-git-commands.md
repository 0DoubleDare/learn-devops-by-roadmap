# Команды в Git

### `git init`

инициализация репозитория, создает скрытую директорию .git хранящая коммиты и историю изменений.
> [!NOTE]
> `git init -b <имя ветки>` - инициализирует репозиторий и задает название главной ветки. В Github-репозиториях с недавнего времени главная ветка называется **main**, но в более старых проектах - **master**
> `git init --bare` - репозиторий для серверов. Он хранит только "сжатые" файлы и соотвественно весят меньше. Обычному разработчики не стоит запоминать эту команду (Только если вы не собирайтесь создавать свой аналог Github / Gitlab )
### `git clone <ссылка на репозиторий>`

клонирует уже существующий репозиторий ( Тот самый bare-репозиторий). Можно кланировать как по **https** для публичных репозиториев как по **ssh** для приватных. Если используется второй вариант (по **ssh**) то вам нужно добавить свой ssh-ключ в настройках вашего профиля.
### `git commit -m "Описание коммита"` 

коммит хранит снимок всех файлов, грубо говоря копию всех файлов с некоторыми оптимизациями для легковестности. Git хранит всю историю о том, когда какой коммит был сделан. 
Если вам нужно написать более подробное описание коммита, введите просто `git commit` (без флага `-m`) откроется текстовый файл хранящий описание файлов которые были изменены / удалены / добавлены.
> [!TIP]
> Хорошей практикой является часто создавать небольшие коммиты для фиксации конкретных изменений. Не стоит раз в несколько дней активной разработки создавать большие коммиты - это ухудшает контроль над проектом и откат изменений. Вашим коллегам будет легче просматривать на условные 50 строк кода реализующих конкретную фукнцию.
### `git branch <имя новой ветки>`

Создает новую ветку с заданным именем. Ветки очень легковесны так как это лишь ссылки на определненный коммит в репозитории
> [!TIP]
> Чтобы визуально просмотреть историю ваших коммитов со всеми сущетсвующими ветками выполните встроенную команду `git log --graph`
> Как и с коммитами, перед тем как делать важные изменения в коде лучше создайте отдельную ветку в вашем локальном репозитории и после добавления фукнционала отправьте pull request (запрос на слияние) на ваш удаленный репозиторий.
### `git checkout <имя ветки>`

Переход к одной из существующих веток
> [!TIP]
> `git checkout -b <имя новой ветки>` - создает новую ветку с заданным именем и сразу переходит на неё. По сути это две команды `git branch <имя новой ветки>` и `git checkout <имя только что созданной ветки>` в одной
### `git merge <ветка>`

Склеивает изменения из заданной ветки создавая новый коммит "слияния" в текущей ветке с двумя родителями. 
К примеру у нас есть ветка main и develop и вы хотите склеить develop и main так, чтобы в ветке main создался новый коммит со всеми изменениями. Для этого:
1. `git checkout main`
	Переходим в ветку в которой хотим создать "смерженный" коммит
2. `git merge develop`
	После этой команды в ветке develop появится новый коммит, содержащий текущие изменения + изменения из main, то есть двух родителей
Если вы хотите чтобы ваши ветки хранили последнее изменение репозитория проверните те же команды:
3. `git checkout develop`
4. `git merge main`
Теперь все ваши ветки (main и develop) содержат последние изменения репозитория
### `git rebase <ветка>`

В отличие от `git merge` который создает отдельный коммит со всеми изменениями из двух веток, `rebase` создает последовательную и линейную историю коммитов соединяя в себе историю выбранных веток. Есть интерактивная версия команды - `git rebase -i <ветка>` где можно выбрать и просмотреть историю коммитов
