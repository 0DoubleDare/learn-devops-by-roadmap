# Git
База не только для Devops, но и для каждого программиста. Лучший способ понять Git это создать свой репозиторий (было неплохо какой-нибудь реальный проект) и оттачивать навыки с контролем версий.
## **Основные определения:**
**Репозиторий ( Repo )** - "хранилище", папка проекта храняща предыдущие файлы и историю изменений помимо текущих.

**Ветка ( Branch )** - Ссылка на конкретный коммит, так как они являются лишь ссылками то весят очень мало. Поэтому ветки нужно создавать часто особенно для совместной работы. Ветки можно: сливать, создавать, переключать.

**Тег ( Tag )** - Метка для ветки, отличия тега от ветки в том что теги никак нельзя менять. Их используют для версиорирования проекта (v0.1-alpha, v1.1-beta, v2.2-release).
## Список команд
`git init` 
	инициализация репозитория, создает скрытую директорию .git хранящая коммиты и историю изменений.

> [!NOTE] Дополнительные команды
> 
> `git init -b <имя ветки>` - инициализирует репозиторий и задает название главной ветки. С недавнего времени главная ветка называется **main**, но в более старых проектах - **master**
> `git init --bare` - репозиторий для серверов. Он хранит только "сжатые" файлы и соотвественно весят меньше. Не нужно сильно углубляться в эту команду (Только если вы не собирайтесь создавать свой аналог Github / Gitlab )

`git clone <ссылка на репозиторий>` 
	кланирует уже существующий репозиторий ( Тот самый bare-репозиторий). Можно кланировать как по **https** для публичных репозиториев как по **ssh** для приватных. Если используется второй вариант (по **ssh**) то вам нужно добавить свой ssh-ключ в настройках вашего профиля.
`git commit -m "Описание коммита"` 
	коммит хранит снимок всех файлов, грубо говоря копию всех файлов с некоторыми оптимизациями для легковестности. Git хранит всю историю о том, когда какой коммит был сделан. 
	Если вам нужно написать более подробное описание коммита, введите просто `git commit` (без флага `-m`) отроектся текстовый файл хранящий файлы которые были изменены / удалены / добавлены.
	
> [!NOTE] Важное примечание
> Хорошей практикой является часто создавать небольшие коммиты для фиксации конкретных изменений. Не стоит раз в несколько дней активной разработки создавать большие коммиты - это ухудшает контроль над проектом и откат изменений. Вашим коллегам будет легче просматривать на условные 50 строк кода реализующих конкретную фукнцию.
